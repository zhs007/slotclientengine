## 3. 数据合同

`bg-bar` 只从 step 0 的 `logic.getStep(0).getComponent("bg-bar")` 读取：

- `features` 必须为长度 5 的数组。
- feature 只允许 `normal | wild | up`。
- `@type` 如果存在，必须为 `type.googleapis.com/sgc7pb.FeatureBar2Data`。
- `usedFeatures`、`cacheFeatures` 必须为数组。
- `curFeature` 必须为已知 feature。
- `basicComponentData.usedScenes / usedOtherScenes / usedResults / usedPrizeScenes / srcScenes / pos` 必须为空数组。
- `basicComponentData.coinWin / cashWin` 必须为 0。

如果本轮没有 `bg-bar` 组件，返回 `null`，空传送带不参与完成条件；如果组件触发但数据非法，显式失败。

## 4. Slot 映射

`slotRectsInConveyor` 表示传送带的视觉格子，不表示 symbol 尺寸；symbol 可以比格子大，运行时以 rect 中心对齐 symbol 中心。live 视觉验收后已按 conveyor 分隔线重新校准，避免把 symbol 尺寸当作格子尺寸。

横屏 `conveyor1.png`：

```text
0 { x: 55, y: 75,  width: 174, height: 126 }
1 { x: 55, y: 204, width: 174, height: 132 }
2 { x: 55, y: 339, width: 174, height: 132 }
3 { x: 55, y: 474, width: 174, height: 132 }
4 { x: 55, y: 609, width: 174, height: 132 }
```

竖屏 `conveyor2.png`：

```text
0 { x: 74,  y: 55, width: 153, height: 115 }
1 { x: 232, y: 55, width: 153, height: 115 }
2 { x: 390, y: 55, width: 153, height: 115 }
3 { x: 548, y: 55, width: 153, height: 115 }
4 { x: 706, y: 55, width: 153, height: 115 }
```

每次 spin 的 shift：

```text
features[0]: slot 3 -> slot 4，播放 win 后隐藏
features[1]: slot 2 -> slot 3
features[2]: slot 1 -> slot 2
features[3]: slot 0 -> slot 1
features[4]: 入口外 -> slot 0，播放 appear
```

完成后的 idle 队列为 `[features[1], features[2], features[3], features[4]]`，仅作为当前视觉状态。后续 live 验收发现服务端连续两轮 `FeatureBar2Data.features` 不保证与上一轮视觉 idle 队列一致，因此下一次 spin 以服务端新下发的本轮 `features` 为权威，客户端重建传送带动画，不因两轮 feature 队列不同而失败。
