# victoryani-demo 相机交互与问题定位任务计划

## 1. 任务目标

对 `apps/victoryani-demo` 做一次可执行的交互优化与问题定位增强，新增“相机式观察”能力，帮助后续稳定复现并判断当前项目里的隐藏 bug。目标不是直接猜测 bug，而是先补齐足够可靠的观察、操作和验证手段。

本计划要求落地以下能力：

- 鼠标拖动时可平移场景，效果类似拖动摄像机观察画面。
- 鼠标滚轮时可缩放整个场景。
- 滚轮缩放只在鼠标位于 Pixi 渲染区上方时生效。
- 为避免误触，只有当 Pixi 渲染区处于“已选中/已激活”状态时，滚轮缩放才生效。
- 调整后仍能保持现有播放、停止、重播、循环等功能可用。
- 给出明确的手工验收与自动化校验方案，帮助后续继续定位 bug。

## 2. 当前实现现状

基于仓库内现有代码，可确认 `apps/victoryani-demo` 当前采用如下结构：

- `src/main.ts`
  - 创建 Pixi `Application`。
  - 创建 `viewportRoot` 与 `sceneRoot` 两层容器。
  - `VictoryPlayer.root` 被挂到 `sceneRoot` 下。
  - 通过 `computeCanvasLayout()` 仅调整 canvas 的 DOM 尺寸和位置，使其按容器居中缩放显示。
- `src/layout.ts`
  - 当前只负责“设计尺寸 -> DOM canvas 展示尺寸”的适配。
  - 没有承担交互式相机状态的职责。
- `src/styles.css`
  - `stage-host` 是画布外层容器，`canvas` 为绝对定位。
  - 目前没有“选中渲染区”“拖拽中”“可缩放”等交互态样式。

这意味着当前项目里至少存在两套缩放语义：

- 一套是 DOM 层面的 canvas 显示缩放，用于适配可视区域。
- 一套是 Pixi 容器树层面的场景变换，目前尚未实现。

后续新增摄像机交互时，必须避免把这两套缩放混在一起，否则非常容易引出以下问题：

- 拖拽位移与实际视觉位移不一致。
- 滚轮缩放倍数受 DOM 缩放和 Pixi 缩放双重叠加影响。
- 缩放后点击命中区域、拖拽距离和鼠标位置换算出现偏差。
- resize 后场景位置被重置或错位。

## 3. 需求解释与默认约束

为确保任务无需额外上下文即可执行，本计划采用以下默认解释：

- “类似控制摄像机”：
  - 用户拖动鼠标时，移动的是场景观察位置，不是修改图层数据。
  - 实现方式默认采用变更 `viewportRoot` 或 `sceneRoot` 的平移与缩放。
- “缩放整个 scene”：
  - 缩放对象默认是承载 `VictoryPlayer.root` 的场景容器。
  - 不修改 `project.json`、不改图层原始坐标。
- “鼠标在 pixi 渲染区上时，可以点选中 pixi 渲染区才生效”：
  - 默认解释为需要先点击一次 `stage-host` 或 canvas，使其进入 active 状态。
  - 仅在 active 状态且滚轮事件来源于该渲染区域时，才响应缩放。
  - 当焦点切走、点击控制面板、或点击渲染区外部时，active 状态可取消。
- 若拖拽与播放控制发生冲突，以“保证场景交互稳定、避免误触文本选择和页面滚动”为优先。

## 4. 完成定义

当以下条件全部满足时，视为任务完成：

- 在 `apps/victoryani-demo` 中实现独立的场景交互状态，不与播放时间线强耦合。
- 鼠标按下并拖动 Pixi 渲染区时，场景能连续平移。
- 鼠标松开、指针离开或窗口失焦后，拖拽状态能可靠结束，不出现“卡住持续拖拽”。
- 鼠标滚轮在 Pixi 渲染区内且渲染区已激活时，可缩放场景。
- 鼠标滚轮在渲染区外、或渲染区未激活时，不触发场景缩放。
- 缩放具有上下限，避免无限放大或缩小导致画面丢失。
- resize 后，场景缩放和平移状态有明确策略：
  - 要么保留用户当前相机状态；
  - 要么按计划中定义的方式重算并重置；
  - 不能出现无规则跳变。
- 播放、停止、重播、循环能力仍可正常工作。
- 新增最小测试或可验证逻辑，覆盖至少：
  - 相机状态计算
  - 缩放边界
  - 激活态门禁
- `README.md` 补充相机交互说明与手工验收步骤。
- 任务完成后新增一份中文任务报告，放在 `tasks/` 下，命名为 `9-victoryani-demo-camera-debug-[utctime].md`，其中 `utctime` 使用 UTC 时间，格式示例：`260403-042525`。
- 若本次执行修改了仓库协作规则、目录规范或基础脚本，需要同步更新根级 `agents.md`；若只改 demo、测试、文档，则无需修改。

## 5. 默认技术方案

### 5.1 相机容器分层

建议明确分成三层职责：

- DOM 适配层：
  - 继续由 `computeCanvasLayout()` 负责 canvas 在 `stage-host` 中的展示尺寸与居中偏移。
- 场景相机层：
  - 新增专门的 camera/viewport 状态，负责 `x`、`y`、`scale`。
- 播放内容层：
  - `VictoryPlayer.root` 继续只负责动画内容，不直接承担交互状态。

建议容器职责如下：

- `app.stage`
  - 只做最外层 Pixi 根容器。
- `viewportRoot`
  - 作为“相机容器”优先候选，统一承接平移和缩放。
- `sceneRoot`
  - 挂实际场景内容，可保持原有语义。

默认优先选择“把平移与缩放都挂在 `viewportRoot` 上”，这样 `sceneRoot` 仍可代表逻辑场景本身，职责更清晰。

### 5.2 交互状态模型

建议新增独立状态对象，例如：

- `isActive`
- `isDragging`
- `dragStartClientX`
- `dragStartClientY`
- `dragStartCameraX`
- `dragStartCameraY`
- `cameraX`
- `cameraY`
- `cameraScale`
- `minScale`
- `maxScale`

这部分建议抽成独立模块，避免全部塞进 `src/main.ts`。

推荐新增文件方向：

- `apps/victoryani-demo/src/interaction/camera-controller.ts`
- 或 `apps/victoryani-demo/src/interaction/scene-camera.ts`

## 6. 具体实施步骤

### 步骤 1：梳理并冻结现有布局职责

执行内容：

- 先确认 `computeCanvasLayout()` 只处理 DOM 展示缩放，不参与交互式场景变换。
- 在计划执行时避免把“滚轮缩放”直接写成修改 canvas CSS 尺寸。

交付标准：

- 代码中能明确区分“DOM 画布适配”和“Pixi 场景缩放”。

### 步骤 2：抽离相机状态与应用函数

执行内容：

- 新建相机控制模块，封装：
  - 默认缩放值
  - 最小/最大缩放
  - 平移值
  - 状态应用函数
  - 缩放钳制函数
- 提供单一入口，将状态同步到 Pixi 容器。

交付标准：

- 主入口文件不直接散落大量鼠标事件中的数值计算。
- 相机状态可以在不依赖 DOM 的情况下做单元测试。

### 步骤 3：实现渲染区激活态

执行内容：

- 给 `stage-host` 或 canvas 增加点击激活逻辑。
- 点击渲染区时进入 active。
- 点击控制面板或渲染区外部时退出 active。
- 为 active/dragging 增加基础视觉反馈，例如边框或阴影变化。

交付标准：

- 用户可以明确判断“当前滚轮是否会控制场景缩放”。

### 步骤 4：实现拖拽平移

执行内容：

- 监听 pointer/mouse 按下、移动、抬起。
- 仅当按下发生在渲染区内时进入拖拽。
- 拖拽中根据鼠标位移更新相机平移。
- 结束条件至少覆盖：
  - `pointerup`
  - `pointercancel`
  - `mouseleave` 或捕获丢失
  - `window.blur`

建议：

- 优先使用 Pointer Events，便于统一鼠标与未来触控扩展。
- 拖拽期间禁用文本选中和页面原生拖拽副作用。

交付标准：

- 拖动画面过程连续，无明显抖动或反向移动。

### 步骤 5：实现滚轮缩放

执行内容：

- 监听 `wheel` 事件。
- 增加双重门禁：
  - 事件目标位于渲染区内；
  - 当前渲染区为 active。
- 缩放值使用统一步进和上下限钳制。
- 默认以场景中心为缩放参考点即可；若实现成本可控，可进一步做“以鼠标所在点为缩放中心”。

默认优先级：

- 第一优先级：先做稳定可用的中心缩放。
- 第二优先级：若时间允许，再升级为基于鼠标位置的缩放中心。

交付标准：

- 滚轮缩放不触发页面滚动。
- 连续缩放后场景不会直接消失或失控。

### 步骤 6：定义 resize 行为

执行内容：

- 明确窗口尺寸变化时，相机状态如何处理。
- 默认建议：
  - 保留 `cameraScale`、`cameraX`、`cameraY`；
  - 仅重新计算 canvas DOM 展示尺寸。

若保留状态导致体验异常，可退而采用：

- 保留缩放值；
- 平移按新 viewport 做一次边界修正或重置。

交付标准：

- resize 后不出现随机跳变。

### 步骤 7：增加调试辅助信息

执行内容：

- 在控制面板增加只读调试信息，至少包含：
  - active 状态
  - 当前缩放值
  - 当前相机平移值
- 如实现成本低，也可增加“重置视角”按钮。

这样可以帮助判断 bug 来源到底属于：

- 原始动画内容；
- 遮罩/混合模式；
- 场景相机状态；
- DOM 适配层。

交付标准：

- 调试数据可视，便于复现问题时记录状态。

### 步骤 8：补齐测试

执行内容：

- 为相机状态模块增加单测，至少覆盖：
  - 缩放上下限钳制
  - 激活态门禁判断
  - 拖拽位移换算
- 如果事件层不适合完整 DOM 测试，至少保证纯函数部分被覆盖。

交付标准：

- `pnpm --filter victoryani-demo test` 能覆盖新增核心逻辑。

### 步骤 9：补全文档与任务报告

执行内容：

- 更新 `apps/victoryani-demo/README.md`：
  - 新增相机交互说明
  - 新增手工验收步骤
  - 写明当前交互边界
- 任务完成后新增任务报告：
  - `tasks/9-victoryani-demo-camera-debug-[utctime].md`

交付标准：

- 不依赖口头说明，后续执行者仅看仓库文件即可理解如何使用和验收。

## 7. 建议验收用例

### 7.1 基础交互

- 打开页面后，点击 Pixi 渲染区，确认出现激活态反馈。
- 激活后滚轮向上/向下，场景可缩放。
- 未点击渲染区前直接滚轮，场景不应缩放。
- 点击控制面板后再滚轮，场景不应缩放。
- 按下并拖动渲染区，场景随鼠标平移。
- 拖拽中松开鼠标后再次移动鼠标，场景不应继续移动。

### 7.2 与播放功能共存

- 自动播放状态下拖拽和平移，动画仍持续播放。
- 点击 `Stop` 后拖拽/缩放仍可操作场景。
- 点击 `Replay` 后相机状态符合预期：
  - 默认建议保留相机状态，不应强制重置。
- 勾选/取消循环不影响相机交互。

### 7.3 bug 判断辅助

- 放大后检查遮罩边缘是否异常偏移。
- 平移后检查混合模式图层是否出现与原位置相关的错误。
- 连续拖拽、缩放、重播后，检查是否出现对象残留、容器错位或滤镜异常。
- resize 浏览器后再次拖拽、缩放，确认状态仍稳定。

## 8. 风险与处理策略

### 风险 1：DOM 缩放和 Pixi 缩放混用导致数学关系混乱

处理策略：

- 明确禁止用修改 canvas CSS 宽高的方式实现滚轮缩放。
- DOM 层只负责适配，场景层只负责交互变换。

### 风险 2：拖拽状态丢失或卡住

处理策略：

- 使用 Pointer Events。
- 覆盖 `pointerup`、`pointercancel`、失焦等收尾逻辑。

### 风险 3：滚轮误触造成页面滚动或控制面板滚动冲突

处理策略：

- 只有 active 且命中渲染区时才 `preventDefault()`。
- 控制面板区域维持原有滚动与交互。

### 风险 4：实现后依旧难以判断 bug 来源

处理策略：

- 补充调试信息和重置视角入口。
- 让“交互状态”和“动画播放状态”都能被观察到。
