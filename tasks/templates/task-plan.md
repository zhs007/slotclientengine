# 任务计划模板

> 规划会话使用本模板创建 `tasks/<id>-<task-name>.md`：
>
> - `<task-name>` 使用小写英文 kebab-case；
> - 只制定计划，不实施、不安装依赖、不运行重型测试、不生成执行报告；
> - 删除占位说明和无关小节，最终计划默认不超过 400 行；
> - 计划不依赖对话内容，但可以精确引用仓库文件，不要全文复制规则、README、manifest 或源码。

# <id> <task-name> 任务计划

## 1. 目标与完成定义

### 目标

简洁说明要解决的问题、使用场景和期望结果。

### 完成定义

- [ ] 用户可观察或自动化验证的最终行为。
- [ ] 必须保持不变的现有行为。
- [ ] 正式交付物、生成物和必要人工验收。

## 2. 范围

### 包含

- 实际需要实现的能力。
- 涉及的 app、package、schema、manifest、生成器或 consumer。

### 不包含

- 容易误解为本任务内容的相邻需求。
- 顺手重构、未来扩展、未提供资源和不需要的兼容路径。

## 3. 制定计划时的基线

```text
UTC:
HEAD:
branch:
git status --short --untracked-files=all:
```

- 区分已有的相关与无关修改；执行时保留用户无关修改。
- 列出实际读取的根 `AGENTS.md`、领域规则和 package 规则，不复制正文。
- 记录当前入口、数据流、public API、schema、测试、配置和能力缺口，使用“结论 + 精确路径 + symbol/字段名”。
- 默认不审计完整 Git 历史。只有合同无法从当前代码、测试和文档确认时，才记录定向历史结论。

## 4. 需求解释与技术决策

### 需求解释

- 把用户需求转换为可实现、可验收的行为。
- 明确已确认事实、合理假设和待确认事项。
- 无法从仓库确认的内容不得编造。
- 如果前期讨论已形成用户确认的《讨论结论》，以最后确认版本为需求合同；被否决方案只能作为非目标，不能重新进入实施范围。

### 关键决策

对重要决策说明：

1. 选择什么；
2. 为什么符合当前架构；
3. 对 public contract 和 consumer 的影响；
4. 为什么不采用会扩大范围或引入 fallback 的替代方案。

简单任务不制造不必要的设计矩阵。

## 5. 职责与合同

按实际任务填写并删除无关项：

- **模块职责**：业务语义、通用算法和 consumer 分别属于哪里。
- **数据/API**：输入、输出、versioned schema、strict validation 和兼容边界。
- **资源生命周期**：owner、prepare、commit、rollback、cleanup、destroy。
- **失败策略**：未知类型、非法状态、缺资源和错误版本尽早显式失败。
- **禁止行为**：重复状态机、第二份资源表、placeholder、路径猜测、静默 alias 或效果降级。

## 6. 文件范围

### 预计新增

```text
path/to/new-file
```

### 预计修改

```text
path/to/existing-file
```

### 原则上不应修改

```text
path/or/package
```

执行时若需明显扩大 public API、lockfile、架构职责或文件范围，必须先说明原因，不能修改计划来事后合理化。

## 7. 实施步骤

每一步写清操作、目标文件和预期结果，不使用“完成全部开发”之类不可执行描述。

1. **确认执行基线**
   - 重新核对 HEAD、工作区和权威文件。
   - 判断仓库变化属于小幅适配还是需要重新规划。

2. **实现核心合同**
   - 写出具体模块、API、schema 或资源变化。
   - 说明本步完成后应成立的行为。

3. **接入 consumer**
   - 明确调用链、配置来源和禁止的重复实现。

4. **同步测试与生成物**
   - 添加或调整直接保护合同的测试。
   - 使用正式生成器更新生成物并运行 checker。

5. **文档与收尾**
   - 按需更新 README 或最小范围领域规则。
   - 运行计划规定的定向验收并生成 UTC 报告。

## 8. 测试与验收

### 测试原则

- 不为错误或过时测试扭曲生产代码。
- 测试期望与明确合同冲突时更新测试；测试暴露真实 bug 时修生产代码。
- 覆盖正常路径、关键边界、strict failure，以及适用的 rollback/cleanup/destroy。
- 已有测试已经证明且本任务不改变的行为，不重复扩张验证。

### 验收级别

选择最低充分级别并说明理由：

- `L0`：文档、注释、无行为变化的版本文本或简单配置。
- `L1`：单 package 定向验收，默认级别。
- `L2`：跨包 public API、共享 schema、生成器、正式交付物或直接 consumer。
- `L3`：根工具链、workspace 配置、lockfile、大规模跨包重构、release 或用户明确要求。

不得仅以“更保险”为理由升级。默认命令不超过 6 条，超过时逐条说明必要性。L1/L2 不默认加入根级 `pnpm typecheck/lint/test/build/format:check`。

### 执行会话必须运行

列出精确可复制命令，例如：

```bash
pnpm --filter <package> typecheck
pnpm --filter <package> test
pnpm --filter <package> build
git diff --check
```

- 生成物变化时加入对应 build/check/parity。
- 失败时先最小化复现，不立即扩大到整仓扫描。

### 人工验收

仅在需要时列出浏览器、Cocos Creator、视觉、profiler、外部服务或真实资源验证。fake runtime、编译或单测不能冒充真实运行环境。

### 独立验收建议

填写：`不需要`、`建议` 或 `必须`，并说明是否涉及：

- 跨包 public contract；
- credential、安全或服务器数据边界；
- resource ownership、异步 transaction 或 destroy；
- 正式 schema、ZIP、生成物或 release。

如需独立验收，只列高风险点和最多 3 条复验命令，不重复执行全部验收。

## 9. 环境与依赖

- Node.js 使用仓库要求的 Node 24。shell 没有 Node 时：

  ```bash
  source /Users/zerro/.nvm/nvm.sh
  nvm use 24
  ```

- 统一使用该环境的 Node 和 pnpm，不强制调版本，不切换 npm/yarn。
- 依赖缺失时：

  ```bash
  CI=true pnpm install --frozen-lockfile
  ```

- 只有下载实际失败后才设置代理并重试原命令：

  ```bash
  export http_proxy=http://127.0.0.1:1087
  export https_proxy=http://127.0.0.1:1087
  ```

- 新增依赖或修改 lockfile 必须说明必要性和影响。

## 10. 生成物、文档与规则

- 生成文件由对应生成器更新，禁止手改；修改 YAML 后同步生成并运行 `--check`。
- 行为或 public workflow 改变时更新对应 README。
- 只有稳定、跨任务职责边界才更新协作规则：
  - 全仓：根 `AGENTS.md`；
  - 领域：`docs/agent-rules/*.md`；
  - package：对应目录 `AGENTS.md`。
- 精确配置、资源清单、版本能力和执行证据不追加到根 `AGENTS.md`。

## 11. 执行报告

规划时不生成报告。执行完成后创建：

```text
tasks/<id>-<task-name>-<utctime>.md
```

UTC：

```bash
date -u +%y%m%d-%H%M%S
```

报告简要记录：

1. 最终实现与实际修改文件；
2. 关键决策和计划偏差；
3. 实际验收命令及结果；
4. 未完成的人工验收；
5. 剩余风险和未完成项。

除 L3 外，不收集无关 coverage、完整历史矩阵、全仓统计或 profiler 数据。

## 12. 风险、假设与待确认

### 风险

- 真实的兼容性、性能、资源、交付或外部环境风险。

### 假设

- 只保留执行所需且有仓库证据的假设。

### 待确认

- 没有则写“无”；能从仓库查明的问题不要留给用户。

## 13. 完成清单

- [ ] 目标和非目标已满足。
- [ ] 实际修改未超范围，或偏差已在报告说明。
- [ ] public API、schema、职责和资源生命周期符合计划。
- [ ] 测试、生成物、README 和规则已按需同步。
- [ ] 指定自动化验收已通过。
- [ ] 自动化与人工验收已明确区分。
- [ ] UTC 中文执行报告已生成。

## 14. 执行会话交接

执行会话应：

1. 读取根 `AGENTS.md`、本计划列出的领域规则和本计划；
2. 核对 Git 基线与工作区；
3. 按计划实现，不重新制定另一套方案；
4. 小幅适配当前实现时在报告记录；
5. 重大范围扩张时先停止说明；
6. 只运行计划规定级别的验收；
7. 完成后生成执行报告；
8. 除非用户明确要求，不 commit、不 push、不创建 PR。
