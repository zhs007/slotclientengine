# 任务前期讨论模板

> 本模板用于在写任务计划前讨论需求和技术方案。
>
> 使用方式：
>
> ```text
> 按 `tasks/templates/task-discussion.md` 与我讨论下面的需求。
>
> 需求：……
> ```
>
> 讨论阶段不创建任务计划、不实施、不修改文件、不安装依赖、不运行测试。只有用户明确确认最终方案后，才进入 `task-plan.md`。

## 1. 讨论角色

你是需求与技术方案讨论者，不是当前任务的执行者。

本阶段目标：

- 理解用户真正要解决的问题；
- 通过只读检查确认当前仓库事实；
- 找出会影响实现的歧义、约束和风险；
- 比较确实存在的方案取舍；
- 与用户形成明确、可冻结的最终方案。

本阶段不得：

- 创建或修改任务计划；
- 修改源码、配置、测试、文档或 Git 状态；
- 安装依赖、运行测试或构建；
- 因为讨论过某个方案就把它视为已确认；
- 为显得完整而扩大到无关系统或未来需求。

## 2. 只读分析范围

1. 读取根 `AGENTS.md`，再按路由只读取与需求直接相关的领域规则。
2. 只读检查直接相关的源码、测试、manifest、YAML、README 和生成脚本。
3. 默认不遍历整个仓库、不读取无关任务、不审计完整 Git 历史。
4. 只有当前合同无法从代码、测试和文档确认时，才定向检查相关历史。
5. 能从仓库查明的问题先自行查明，不把它们作为问题抛给用户。

## 3. 第一轮回复

第一轮先给出：

### 需求理解

- 用自己的话复述用户目标。
- 区分目标、实现手段和用户可观察结果。
- 指出可能存在的误解。

### 当前实现

- 只列与决策有关的仓库事实。
- 使用精确路径、symbol、配置字段或 public API 作为证据。
- 不复制大段源码、规则或历史。

### 需要讨论的问题

- 只提出会实质改变方案、范围、兼容性或验收的问题。
- 优先一次讨论一个关键决策，避免一次抛出大量低价值问题。
- 给出推荐选项和理由，不把所有判断都交给用户。

如果需求已经足够明确，可以直接提出推荐方案，但仍需等待用户确认。

## 4. 方案比较

只有确实存在不同取舍时才使用方案比较。

每个候选方案说明：

- 核心做法；
- 对当前架构和职责边界的影响；
- public API、schema、资源和 consumer 影响；
- 优点；
- 风险和代价；
- 验收难度；
- 是否推荐。

不要列明显错误或无意义的“陪跑方案”。优先推荐最小、符合现有架构且容易显式失败的方案。

## 5. 讨论原则

- 当前代码、测试、manifest、YAML 和明确规则优先于猜测。
- 未确认内容标记为“假设”或“待确认”。
- 不把未知类型、缺资源、错误版本或非法状态设计成静默 fallback。
- 不为满足过时测试扭曲生产合同。
- 不因为未来可能需要就提前增加扩展层、兼容层或第二套数据源。
- 讨论中被否决的方案不再进入实施范围，除非用户后来明确恢复。
- 用户改变需求时，更新当前理解并明确哪些旧结论失效。

## 6. 方案冻结条件

只有同时满足以下条件，才建议结束讨论：

- 用户目标和非目标明确；
- 最终方案已被用户明确确认；
- 模块职责和主要数据流明确；
- public contract、兼容性和失败策略明确；
- 预计影响范围可以界定；
- 自动化与人工验收边界明确；
- 没有会实质改变实现的未决问题。

不要把“用户没有反对”视为确认。需要用户明确表达“确认”“采用这个方案”或等价意思。

## 7. 《讨论结论》格式

方案满足冻结条件后，整理以下内容并停下来等待用户确认：

```markdown
# 讨论结论

## 已确认目标

- ...

## 最终采用方案

- ...

## 职责与数据流

- ...

## 范围

### 包含

- ...

### 不包含

- ...

## Public contract 与兼容性

- ...

## 失败与 fallback 边界

- ...

## 验收建议

- 建议级别：L0 / L1 / L2 / L3
- 自动化重点：...
- 人工验收：...
- 是否建议独立验收：不需要 / 建议 / 必须

## 被否决方案

- 方案：...
- 原因：...

## 假设与待确认

- 无 / ...
```

结论只包含最终有效信息，不记录冗长对话过程。

## 8. 转入任务计划

用户确认《讨论结论》后，应等待用户明确要求写计划。推荐指令：

```text
方案确认。

以我最后确认的方案和你整理的《讨论结论》为准，
按 `tasks/templates/task-plan.md` 编写任务 <id> 的计划，本轮不执行。

讨论过但未采纳的方案只能记录为非目标或被否决方案，不能进入实施范围。
```

编写计划时：

- 以最后确认的《讨论结论》为需求合同；
- 再做必要的只读基线核对；
- 不重新打开已冻结的设计问题；
- 如果讨论结论与当前仓库事实冲突，指出冲突并停止，不自行改变已确认方案；
- 最终计划仍必须不依赖对话内容即可执行。
