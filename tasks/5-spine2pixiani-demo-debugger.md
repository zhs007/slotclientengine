# spine2pixiani-demo 调试交互增强任务计划

## 1. 任务目标

在现有 `apps/spine2pixiani-demo` 基础上补齐一套可直接用于排查 Spine 数据问题的调试交互能力，具体包括：

- 鼠标拖动画布时可移动“摄像机”视角。
- 鼠标滚轮可缩放整个场景。
- 左侧新增鼠标模式切换，至少支持：
  - 点选
  - 移动
- 点选模式下可选中节点，并显示节点名称。
- 页面右侧显示节点树状结构，便于定位骨骼层级、插槽归属和异常节点。

本计划为可直接执行版本，不依赖额外口头说明。

## 2. 当前现状

基于当前仓库代码，可确认以下事实：

- 页面入口在 `apps/spine2pixiani-demo/src/main.ts`。
- 当前页面结构只有两栏：
  - 左侧控制面板 `createAnimationSelect(...)`
  - 中间舞台区域 `stageShell`
- 场景主节点当前直接挂在 `app.stage -> stageRoot -> CabinAnimationEntity`。
- 现有缩放仅用于初始化时给 `cabinEntity` 设定适配舞台的固定比例，不支持运行时缩放。
- 现有代码没有单独的“摄像机容器”或“视口控制器”概念。
- `CabinScene` 内部已经维护了：
  - `boneNodes: Map<string, Container>`
  - `slotNodes: Map<string, Sprite>`
- `createBones()` 中已为 bone 节点设置 `node.name = bone.name`，说明骨骼节点命名基础已具备。
- 当前没有：
  - 节点点击命中逻辑
  - 节点选中高亮逻辑
  - 调试树数据结构
  - 左侧鼠标模式切换 UI
  - 右侧调试面板 UI

换句话说，这次任务不是从零做 demo，而是在已有播放 demo 上补一层“调试视图 + 视口交互”。

## 3. 完成定义

当以下条件全部满足时，可认为本任务完成：

- 页面布局升级为三栏或等价结构：
  - 左侧控制区
  - 中间舞台区
  - 右侧节点树区
- 鼠标模式可在“点选 / 移动”之间切换，且当前模式在 UI 上明确可见。
- 移动模式下，按下鼠标拖动舞台可以平移摄像机视图，不影响动画继续播放。
- 鼠标滚轮可以缩放整个场景，并且缩放有最小值和最大值限制，避免缩得过小或过大。
- 点选模式下，点击舞台中的可识别节点后：
  - 能记录当前选中节点
  - 左侧或右侧能看到节点名称
  - 节点树中对应项同步高亮
- 右侧节点树至少能展示：
  - bone 层级关系
  - 每个 bone 下关联的 slot 名称或可识别挂载信息
- 节点树数据与当前加载的 `cabinAnimationData` 一致，不依赖手工硬编码单独一份树。
- 基础交互在 `cabin` 与 `cabin_s` 两个动画下都可正常使用。
- 至少补充一组测试或可自动校验的纯函数验证，覆盖新增调试数据结构或视口状态逻辑。
- `README.md` 如已不足以说明新交互，需要同步补充使用说明。
- 任务完成后，在 `tasks/` 下新增中文任务报告，命名为 `5-spine2pixiani-demo-debugger-[utctime].md`，其中 `utctime` 使用 `年月日时分秒` 格式，例如 `260402-153000`。
- 若本次实现改变了仓库协作规则、目录规范或基础脚本，再同步更新根级 `AGENTS.md`；若只是应用功能增强，则无需修改。

## 4. 默认实现假设

为保证任务可落地，默认按以下假设推进：

- “摄像机”不引入真正的 3D Camera 概念，而是通过额外的 Pixi `Container` 包装场景，实现平移与缩放。
- 点选优先支持 bone 节点与 slot 节点两类调试对象；若 Pixi 精确命中成本过高，可先基于调试代理图元或包围盒实现。
- 节点树的核心用途是排查数据错误，因此优先保证“名称、父子关系、slot 归属、当前选中同步”正确，不要求首版就支持树节点拖拽、搜索、折叠记忆等高级功能。
- 调试 UI 继续沿用当前 demo 的页面风格，不引入额外 UI 框架。
- 首版允许把“选中节点名称”展示在左侧控制面板，并在右侧树中同步高亮；如果实现成本低，可额外显示节点类型和父节点。
- 视口缩放作用于整个 scene 根节点，而不是只缩放单个骨骼实体。
- 鼠标滚轮缩放以桌面浏览器为主场景，移动端手势支持不在本次范围内。

## 6. 关键设计决策

### 决策 1：摄像机能力的落点

建议新增一层明确的场景层级：

- `app.stage`
  - `viewportRoot`
    - `sceneRoot`
      - `CabinAnimationEntity`

建议含义如下：

- `viewportRoot` 负责摄像机平移和缩放。
- `sceneRoot` 保留业务场景原点和默认摆位。
- `CabinAnimationEntity` 继续只负责动画播放，不直接承担输入交互职责。

这样可以避免把“动画缩放”和“用户视角缩放”混在同一个容器上。

### 决策 2：点选与移动模式互斥

建议鼠标模式使用显式互斥状态，而不是用修饰键临时切换。

建议模式：

- `select`
- `pan`

原因：

- 用户要排查节点数据时，需要明确当前点击到底是“选中节点”还是“拖动画布”。
- 显式模式更适合后续继续扩展，例如增加“框选”“测量”等调试模式。

### 决策 3：节点树的数据来源

建议节点树直接从 `SpineModel` 派生，而不是从 DOM 或 Pixi DisplayObject 反推。

树中建议至少包含：

- `id`
- `name`
- `type`，例如 `bone` / `slot`
- `parentId`
- `children`
- 关联 slot 列表或调试描述

原因：

- `SpineModel` 是当前 demo 唯一稳定的数据源。
- 这样可以避免动画每帧更新时树结构抖动。
- 也更容易补测试。

### 决策 4：点击命中的实现策略

建议首版优先采用“调试命中代理”的实现，不直接依赖每个 sprite 的像素命中。

可落地方向：

- bone 节点：在调试层为每个 bone 放置一个透明但可交互的小圆点或小十字标记。
- slot 节点：如有必要，为 attachment 当前世界坐标生成简单包围盒用于点选。

默认优先级：

1. 先保证 bone 可点选。
2. 再补 slot 可点选。
3. 如果 slot 精准命中复杂，可先在树上支持点击选中 slot，舞台点击首版仅支持 bone。

### 决策 5：树与舞台的双向同步

建议建立一个统一的选中状态，而不是分别在 UI 和场景中各自维护。

建议统一状态字段至少包含：

- `selectedNodeId`
- `selectedNodeName`
- `selectedNodeType`
- `mouseMode`
- `zoom`
- `panX`
- `panY`

这份状态可以先用轻量对象 + 事件回调维护，不必引入状态库。

## 7. 任务拆分

### 任务 1：补齐调试交互的页面骨架

目标：先让页面具备容纳“左侧模式控制 + 中间舞台 + 右侧节点树”的结构。

执行内容：

- 在现有 `createAnimationSelect` 基础上扩展左侧控制面板。
- 新增鼠标模式切换区域，至少包含：
  - 点选
  - 移动
- 新增选中节点信息展示区域，至少显示：
  - 当前模式
  - 当前选中节点名
  - 节点类型
- 在主布局中新增右侧调试树容器。
- 调整 `styles.css`，使桌面端为三栏布局，窄屏时自动折叠为纵向堆叠。

验收标准：

- 页面结构稳定可见，不遮挡现有舞台。
- 在没有选中节点时，调试信息区域有明确空状态提示。

### 任务 2：引入视口容器与摄像机状态

目标：把用户交互缩放/平移从动画实体本身中解耦出来。

执行内容：

- 在 `main.ts` 中新增 `viewportRoot` 和 `sceneRoot` 两层容器。
- 把当前初始化时给 `cabinEntity` 的默认位置、默认缩放，整理为：
  - 业务默认摆位
  - 用户运行时视口偏移/缩放
- 提取视口状态管理逻辑，建议封装为独立模块或局部控制器函数。
- 定义视口约束：
  - `minZoom`
  - `maxZoom`
  - 默认 `zoom`
  - 默认 `pan`

验收标准：

- 在不接入鼠标交互前，场景视觉效果与当前 demo 基本一致。
- 后续平移和缩放只影响 `viewportRoot`，不破坏动画播放。

### 任务 3：实现移动模式下的拖拽平移

目标：支持用鼠标拖动舞台，移动当前观察区域。

执行内容：

- 在舞台容器或 canvas 上注册 pointer 事件。
- 当鼠标模式为 `pan` 时：
  - `pointerdown` 记录起始位置
  - `pointermove` 按位移更新视口平移
  - `pointerup` / `pointerupoutside` 结束拖拽
- 为舞台和页面增加合适的 cursor 表现，例如：
  - `select` 模式显示默认光标
  - `pan` 模式显示 `grab` / `grabbing`
- 避免拖动时误触发节点选中。

验收标准：

- 动画播放过程中可连续拖动画布。
- 切回点选模式后，不会残留拖拽状态。

### 任务 4：实现滚轮缩放整个场景

目标：支持快速放大缩小，便于排查细节。

执行内容：

- 监听舞台区域 `wheel` 事件。
- 以统一步进或平滑系数更新 `zoom`。
- 对 `zoom` 做上下限钳制。
- 默认以鼠标所在点为参考进行缩放；若实现成本过高，首版可先以舞台中心为参考，但需要在报告中说明。
- 处理浏览器默认滚动，避免页面跟随滚动。
- 在左侧或右侧显示当前缩放比例，便于调试。

验收标准：

- 连续滚轮缩放稳定，无明显跳变。
- 缩放不会导致场景瞬间消失或翻转。

### 任务 5：构建节点调试树数据

目标：让右侧树视图有稳定、可测试的数据输入。

执行内容：

- 从 `SpineModel.bones` 构建 bone 树。
- 把 `SpineModel.slots` 作为对应 bone 的子节点或关联项挂到树上。
- 为树节点生成稳定 id，建议规则例如：
  - `bone:${boneName}`
  - `slot:${slotName}`
- 输出适合 UI 渲染的结构体，例如：
  - `name`
  - `id`
  - `type`
  - `children`
  - `meta`
- 将该逻辑放入纯函数模块，便于测试。

验收标准：

- `cabinAnimationData` 能稳定生成完整树结构。
- tree 节点总数与骨骼/插槽数据规模一致，不漏主干节点。

### 任务 6：实现节点树面板与树上选中

目标：先让用户即使不点舞台，也能从树中定位问题节点。

执行内容：

- 新增右侧树组件，递归渲染节点结构。
- 支持点击树节点触发选中。
- 选中后同步更新调试信息区域。
- 对当前选中节点做高亮。
- 如树节点较多，支持基础滚动；首版可不强制实现折叠。

验收标准：

- 用户可以只通过右侧树完成节点定位。
- 树节点点击后，选中状态稳定，不会因动画帧更新丢失。

### 任务 7：实现舞台点选与场景高亮

目标：让“点选模式”真正能从舞台上直接找到节点。

执行内容：

  - 外框
  - 圆点描边
  - 颜色变化
- 树选中和舞台选中保持双向同步。

验收标准：

- 至少 bone 节点可以通过舞台点击选中。
- 选中某个节点后，能在舞台和树中同时看到反馈。

## 8. 实施顺序建议

建议按以下顺序落地，降低返工：

1. 先改页面骨架与三栏布局。
2. 再加 `viewportRoot`，先让场景结构稳定。
3. 再做拖拽平移和滚轮缩放。
4. 之后抽 `debug-tree` 纯函数并把右侧树跑起来。
5. 最后接入舞台点选和高亮。
6. 收尾时补测试、README 和任务报告。

这样做的原因是：

- 视口容器一旦定型，后面的拖拽、缩放和点选都能围绕同一套坐标体系展开。
- 树和选中状态先落地，可以减少舞台命中实现时的调试成本。

## 9. 风险与对应策略

### 风险 1：直接点 sprite 命中不稳定

原因：

- 当前 slot sprite 会每帧变化贴图、位置和旋转。
- 直接依赖纹理像素命中不利于维护。

应对：

- 首版把舞台点击聚焦到 bone 调试代理。
- slot 选中优先通过右侧树支持。

### 风险 2：缩放与初始适配互相干扰

原因：

- 当前 `cabinEntity.scale` 同时承担了“内容适配舞台”的职责。

应对：

- 明确拆成“业务默认 scale”和“用户视口 zoom”两层。

### 风险 3：树结构和场景节点命名不一致

原因：

- 当前 `bone` 与 `slot` 的显示对象并没有统一公共调试 id。

应对：

- 引入稳定 id 规则，并在场景节点和树节点两侧共用。

### 风险 4：交互逻辑散落导致后续难维护

原因：

- 如果把模式、选中、缩放、拖拽分别写在多个 UI 文件里，容易失控。

应对：

- 尽早建立统一调试状态对象和清晰的事件流。
