# Agents

本仓库是基于 `pnpm` + `turbo` 的 TypeScript monorepo，面向 slot 游戏前端引擎开发，后续会承载游戏模版、网络库、渲染库及可运行应用。

## 仓库约束

- Node.js 版本要求：`>=24.0.0`
- 包管理器：`pnpm`
- 构建编排：`turbo`
- 前端构建：`vite`
- 测试框架：`vitest`
- 代码检查：`eslint`
- 代码格式化：`prettier`
- VS Code 调试运行 TypeScript：`ts-node`

## 目录约定

- `apps/`：可运行项目
- `packages/`：内部依赖库
- `packages/gameframeworks`：后续 slot 游戏默认 facade，整合 UI、网络和逻辑数据流。
- `packages/gameloading`：通用轻量 loading 页面、资源加载和 `99%/100%` 生命周期编排；游戏 app 只配置资源列表和回调，不在 app 内复制通用 loading 调度。
- `apps/popupeditor`：纯前端获奖庆祝 popup 编辑器；从第一版使用 logical resource、import review、完整 SHA-256 content-addressed owned payload，并只输出 strict `award-celebration` popup package。普通 popup 不在当前范围。
- `packages/rendercore/popup` 拥有 popup manifest/parser、精确传递闭包、image/VNI/official Spine/image-string layer、五档 BigInt threshold sequence、金额格式、点击/dismiss/end drain 和 runtime snapshot；editor、gamelayouteditor 与 game app 不得复制这些算法。
- `apps/gamelayouteditor` 拥有通用游戏模式 draft、mode 到独立 per-variant 稳定 background / symbols package / award-celebration popup 的显式绑定、独立有向转场工作区、symbols/popup dependency library 与 popup per-variant viewport-center root placement；稳定 Spine 背景只使用显式 single loop，相同 logical resource 跨 mode 只共享 bytes、不共享 node/player/placement。`packages/rendercore/scene-layout` 拥有 strict gameModes/plural symbolPackages/directed transition schema、transition Spine exact event occurrence 与精确资源闭包、固定顶层 `scene-transition-overlay`、目标完整 scene 预准备、Spine event 边界原子切 background/reel/displayed mode、once completion settle、当前模式 popup lifecycle 与 production API。缺显式有向边时不得瞬切、反向复用、自动寻路或回退旧 node state-machine；shared code 不得硬编码 BaseGame/FreeGame/BonusGame、BG/FG、animation/event 名或游戏业务字段。
- `gamelayouteditor` 把 popup ZIP 当自包含 dependency，每个 active variant 只配置相对 viewport center 的 root `x/y/scale`，最终 vendor 到 layout ZIP；popup 内部坐标、tier、layer 和资源只回 popupeditor 编辑。真实获奖 ImgNumber 未提供前，game002/game003 仍保留当前 production win-amount 路径，不得用 CN digits、字体或 fixture glyph 冒充迁移完成。
- `packages/vnicore`：Pixi.js v8 VNI 动画核心库，供 `apps/anieditorv5viewer` 等 Pixi 运行时使用；不要与 `packages/anieditorv5runtime-cc` 的 Cocos Creator runtime 混用。
- `tasks/`：任务计划、任务报告和执行记录
- `docs/`：项目文档

## 执行约定

- `game002` 的消除级联组件名、step 映射和 symbol 特例属于 `apps/game002`；`packages/rendercore` 拥有通用 manifest-driven emphasis/win/remove、非中奖格与其 symbol 本体同步压暗及渐变恢复、可注入 remove/drop predicate，以及 grid-cell unified fall / existing-only dropdown / effect sweep / selective refill spin。game002 的强调段固定为 `0.1s` 渐暗、`1s` 保持、`0.1s` 渐亮；压暗开始的同一边界必须并行启动全部中奖 symbol 的 opening win，强调结束后再按稳定组顺序等待已启动的 win 完成并依次 remove，不能重播 opening win。game002 必须通过函数声明 `WL` 不 remove、不 drop；rendercore 不得硬编码 wild。非期待 initial spin 中 `WL/CN` 都保持全亮；第 2 个真实落地 WL 打开期待 gate 的同一边界起，只剩 `WL` 保持全亮，CN 与其它实际滚动 occurrence 一样压暗；期待 selective refill 从起播边界起就使用“仅 WL 全亮”状态。被压暗 occurrence 的格底使用随 reel slot 同速移动的半透明黑层，symbol 同步使用灰阶 tint 变黑且 alpha 保持 `1`，当前暗度只从 `assets/game002-s3/reel.manifest.json` 的 `spin.dimmingAlpha=0.5` 读取；cascade 非中奖格压暗同样固定为 `0.5`。暗层不得固定在棋盘坐标形成矩形断层，也不得退回奇偶棋盘格；rendercore 只接收通用 symbol-code resolver 和 activation 布尔状态，不得硬编码 `WL/CN`。普通 spin 回弹力度只从同一 reel manifest 的 `spin.bounceStrength` 读取，当前 game002 为 `0`、完全不回弹；rendercore 通用 parser/runtime 以 `1` 表示原始力度并按非负值缩放，app 不得再硬编码第二份数值。普通 symbol 可穿过固定 WL 下落，叠放只服从 manifest `renderPriority`，当前 game002-s3 仅 WL 为 `1`、其它为 `0`。非期待局的 unified fall 不走 spin/appear，并同时保留列内错峰和从左到右列延迟；期待局则严格执行 existing-only dropdown -> refill-hole Nearwin2 sweep -> selective refill spin/appear，survivor 不重新 spin/appear。normal 与 dropdown 实际 animation resource/playback 完全一致时，RenderSymbol 必须保留 player 和时间轴，不能 reset/replay 等价 Loop；不同动画、reset 或 value/tier 变化仍正常切换。fall 期间只给整个 grid-cell reel set 启用一个完整轮子矩形 mask，活动 symbol 自身不得再叠加 mask。CN 数字仍随 occurrence 下落；只有 refill 新增 CN 时才要求当前 step 触发 `bg-gencoins` 并使用服务端值，没有新增 CN 时该组件允许缺失且既有 CN value 必须从 dropdown occurrence 严格携带。全部 step 结束后才播放 global win-amount；win-amount 播放期间仍必须逐帧更新 reel runtime，不能冻结 CN/其它 symbol 的 normal Loop。不得在 app 复制 Pixi/Spine/cascade 算法或增加 fallback。
- `game002` 普通 initial grid-cell spin 不播放任何 Nearwin effect；第 2 个按真实 landing order 落地的 paytable exact `WL` 在同一边界激活期待，该格本身也不补播 effect，后续格才各播放 `Nearwin1.Loop` 真实 1 次。第一枚后续格从 activation boundary 起等待 `800ms` 落地，Nearwin1 在 activation 后约 `133.3333ms`、即 landing 前单次真实时长起播；之后每隔 `100ms` 起播下一格 Nearwin1，每格真实单循环完成即落地，因此 effect start 与 landing cadence 均为 `100ms`。非期待 unified refill 完成时，如果新增的 paytable exact `WL` 使当前盘面按 refill movement 真实完成顺序达到 2 个 WL，也必须在该完成边界激活期待；不倒放已完成的当前 refill，后续 cascade/refill 立即改走期待路径。期待状态跨本轮 win/remove/cascade/refill/global win-amount 保留，只在下一次合法 initial spin 真正开始、apply initial state、fatal cleanup 或 destroy 时清除。期待 refill sweep 固定按 `y desc/x asc`、`80ms` stagger 并使用 Nearwin2 一次；selective spin 的实际起点是最左侧有 hole 列中的最下 hole，按“本列向上层级 + 有 hole 的列从左向右层级”形成右上对角波，同一层同时以 `16ms` start group 边界启动，Nearwin1 与 landing 的稳定顺序按波层、层内从左向右保持 `100ms` cadence。每个 refill 格使用 Nearwin1 真实 1 次，`settleAfterLastStartMs` 为 `800ms`，effect 仍在各自 landing 前单次时长才起播。全部资源、loopCount、timing、order 和 transform 只从 `assets/game002-s3/reel.manifest.json` 读取；rendercore 只接收通用 effect/gate/start group，不得出现 WL、Nearwin 文件名或 game002 语义。
- `game002` 的 grid-cell initial spin 与期待 selective refill spin 每次都必须重新生成逐列本地轮带视觉相位：rendercore 对每列完整公开轮带做 partial Fisher-Yates，为同列各格无重复抽取 phase；只洗 phase、不洗 symbol 顺序。game002 只注入独立本地 `spinPhaseRandom`，production 默认使用 Web Crypto，不得消费服务器 randomNumbers、全局 `Math.random` 或 CN presentation random。最终服务端 scene 仍只覆盖本轮临时 strip 的落点窗口，方向、速度、timing 和 live 公共轮带边界不得改变。
- `game002` cascade 临时汇总使用 cash 单位且为 0 时隐藏；每组严格从 result 的 `cashWin64 -> cashWin` 字段存在性选择 positive safe integer cents，并统一通过 `formatServerUsdAmount` 除以 `100` 显示。普通组在压暗开始、win 请求的同一阶段并行累计整组 cash；CN collect 仍逐格读取当前 occurrence 的 positive safe integer raw coin value，所有 CN item coin 之和必须精确等于 result coin amount，再按 `itemCoin/groupCoin * groupCash` 精确分配 cents，不能整除时显式失败。manifest `cascadeWinPresentation.order/mode/state` 决定稳定排序和 group/sequential collect 编排：当前普通 symbol/WL 为 `order=0`，CN 为 `order=1`；压暗开始时全部普通组直接请求 `win`，CN 组同时请求全部实例的 `Win_Start`，完成后即进入 `Win` loop，之后按 `y` 再 `x` 行优先执行逐格 Collect。四档 CN 官方动画均为 `Collect=0.3333333s`、`End=0.5s`，相邻 Collect 固定以 `0.3s` 起播 cadence 推进，不等待前一枚 Collect/End/release，允许前后段重叠；每枚自身仍严格 `Collect once -> End once -> release`，全部 active item 和 summary 完成后才结束 coin group。official Spine loop 必须上报真实 loop boundary，不能让 pending Collect 永久卡住。服务端允许 WL 参与 CN coin 中奖判定；game002 必须把它作为显式批准的 sequential companion，在全部 CN `Win_Start` 的同时播放 WL 自身 `win`，等待该 win 完成后让 WL 回 normal，但 WL 不贡献 item value、不进入 loop/Collect/End、不 remove/drop。其它 presentation 混用仍显式失败。summary 跨 step/fall 保留，全部 cascade 后、global cash win-amount 前清零隐藏。`packages/rendercore` 拥有 generic counter、排序、state snapshot、cadence 起播、并发 active item drain、逐格 release、Spine loop completion 和 manifest-derived additional state preset；不得硬编码 `CN/WL/bg-win/game002` 或实例 state/animation name。`apps/game002` 只拥有组件、cash/coin/result/value/companion resolver、formatter、cadence、样式与 layout，也不能直接操作 Pixi child、Spine track 或 tier player。
- value-presentation symbol 收到 `spinBlur/disabled` 等带显式 reel state texture 的等价请求时，显式 texture 必须优先于 normal active Spine；tier player 异步 init 无论早到或晚到都不能把当前 reel texture 隐藏成空格。回到 normal/activeSpine 后才显示同一 tier player。normal 与 loop state 复用相同 active Spine resource/playback 时必须在不 replay 的前提下同步新的 semantic playback，使真实 loop completion 能推动 pending state。

- `packages/rendercore` 同时拥有命名 image-string symbol node 和 value-tier ImgNumber 的 string API、nested exact closure、共享资源池及 official Spine slot attach/detach 生命周期；`apps/symbolseditor` 只拥有 standalone dependency 导入、typed draft/UI、ZIP vendoring 与预览输入，不复制 glyph layout、Spine player 或 attach 算法。`apps/gengameconfig` 与 `packages/logiccore` 拥有通用命名数值权重表生成、严格解析和只读查询，不硬编码 CN、coin、组件名或游戏语义；`apps/gamelayouteditor` 的 per-symbol otherScene mapping、权重抽样与 fixed value 只属于 session preview，不进入 layout manifest/ZIP。game002 CN 已迁移到 manifest-driven per-tier ImgNumber，不保留 legacy 完整图片双写；缺 glyph、slot、table、resource、binding 或非法 value 一律显式失败，禁止 fallback。
- `packages/rendercore/scene-layout` 拥有 gamelayouteditor layout manifest 的通用 image-string node、稳定 Spine loop、directed top-overlay Spine event 转场、plural symbols/popup 传递 dependency loader、mode-aware background visibility、完整目标 scene 原子提交与 active standard/grid-cell reel prepare/swap runtime；相同 symbols binding 的 mode 切换默认保留 reel/scene/player，只有显式 `recreateReel` 才重建。`apps/gamelayouteditor` 只拥有 logical resource/draft/转场配置 UI/production preview/ZIP IO，game app 只提交目标 mode 及运行期 scene/local phase。底层 scene node state-machine 可供独立 consumer 使用，但不得作为 Game Layout Editor 或 `requestGameMode()` 的隐藏入口/fallback；不得在 editor/app 复制 event drain、image-string、official Spine player、background visibility 或 reel placement/runtime，也不得把服务器真实轮带写入 layout package。
- `Imgnumbereditor`、`symbolseditor`、`gamelayouteditor` 的上传统一使用 `browserartifactio` 的 kebab-case logical id、bounded source index、Web Crypto 完整 SHA-256 和 flat allocator；上传不自动绑定 glyph/state/node/background/placement。owner payload 固定为 `assets/<64-char-sha256>.<canonical-ext>`，manifest/Spine atlas/VNI refs 必须结构化同步改写；exact content path 可复用但 logical identity 不合并。nested dependency 自包含，consumer 只 vendor 精确闭包、不重新 hash。
